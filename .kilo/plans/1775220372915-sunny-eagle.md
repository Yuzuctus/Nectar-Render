# Audit du Projet Nectar Render (MD-TO-PDF)

> **Date de l'audit :** 3 avril 2026
> **Version du projet :** 0.3.0.dev0
> **Périmètre :** Sécurité, qualité du code, refactoring, maintenance
> **Méthodologie :** Analyse statique manuelle du code source Python, revue des dépendances, examen de la configuration CI/CD, évaluation des pratiques de développement
> **Limite :** Aucun code n'a été modifié. L'audit se base sur la lecture des fichiers sources et de configuration. Aucune exécution de tests ou d'outils d'analyse automatisée n'a été réalisée dans le cadre de cet audit.

---

## 1. Synthèse Exécutive

Nectar Render est une application de bureau Python (Tkinter) qui convertit des fichiers Markdown en PDF et HTML stylisés. Le projet est globalement **bien structuré** avec une séparation claire des responsabilités, une couverture de tests solide et une bonne gestion des erreurs.

**Points forts identifiés :**
- Architecture modulaire avec séparation UI / conversion / services / utilitaires
- Validation et assainissement rigoureux des entrées CSS (injection CSS bien défendue)
- Tests unitaires couvrant les cas limites et les entrées malveillantes
- Pipeline CI/CD fonctionnel (GitHub Actions) avec tests multi-plateformes et multi-versions Python
- Import différé des dépendances lourdes (WeasyPrint, Tkinter)

**Risques principaux identifiés :**
- **Sécurité :** L'assainissement HTML (bleach) est **désactivé par défaut**, permettant l'injection de balises `<script>` via le Markdown
- **Sécurité :** `install_dependencies.bat` exécute des installateurs tiers avec sortie supprimée, masquant d'éventuelles erreurs
- **Qualité :** Code mort dans `cli.py` (affectation sans effet lignes 152-154)
- **Maintenance :** Pas de verrouillage de dépendances (`requirements.lock` / `uv.lock`)
- **Maintenance :** Pas de configuration Ruff dans `pyproject.toml` (règles non définies)

**Note globale :** 7/10 — Le projet est dans un état sain mais présente des axes d'amélioration ciblés, principalement en sécurité par défaut et en rigueur de configuration.

## 3. Findings et Risques par Domaine

### 3.1 Sécurité

#### S-01 : Sanitisation HTML désactivée par défaut — **CRITIQUE**

**Fichiers concernés :**
- `src/nectar_render/converter/markdown_parser.py:278` — `sanitize_html: bool = False`
- `src/nectar_render/config.py:68` — `sanitize_html: bool = False` dans `StyleOptions`

**Description :** Le paramètre `sanitize_html` est `False` par défaut. Cela signifie que tout HTML brut présent dans le Markdown (y compris `<script>`, `<iframe>`, `<object>`) passe directement au rendu PDF/HTML. Pour une application desktop, le risque est limité, mais un fichier Markdown malveillant pourrait exécuter du JavaScript dans le navigateur lors de la prévisualisation HTML.

**Recommandation :** Activer `sanitize_html=True` par défaut. Si certains utilisateurs ont besoin de HTML brut, ils peuvent le désactiver explicitement.

**Impact si non corrigé :** Un fichier Markdown contenant `<script>alert('xss')</script>` exécuterait ce script dans le navigateur lors de la prévisualisation.

---

#### S-02 : Configuration VSCode avec auto-approbation de commandes dans le dépôt — **ÉLEVÉ**

**Fichier concerné :** `.vscode/settings.json`

**Description :** Le fichier `.vscode/settings.json` contient des règles d'auto-approbation pour des commandes shell spécifiques, incluant des chemins absolus vers la machine d'un développeur (`C:/Users/sammy/...`). Ce fichier est ignoré par `.gitignore` (ligne 21), mais il est présent dans le dépôt actuel, ce qui suggère une incohérence de configuration.

**Recommandation :** S'assurer que `.vscode/settings.json` est bien dans `.gitignore`. Si ce fichier doit être partagé, supprimer les règles d'auto-approbation et les chemins personnels.

---

#### S-03 : Installation automatique de dépendances système avec sortie supprimée — **MOYEN**

**Fichier concerné :** `install_dependencies.bat` (lignes 55, 62, 132, 143)

**Description :** Les commandes `winget` et `choco` sont exécutées avec `>nul 2>&1`, masquant toute erreur ou message d'avertissement. De plus, le script installe automatiquement des packages système (MSYS2, Pango, qpdf) sans confirmation explicite de l'utilisateur.

**Recommandation :** Ajouter une option `--silent` vérifiable ou afficher un résumé post-installation. Conserver les logs dans un fichier temporaire consultable.

---

#### S-04 : Cache d'index d'images sans limite de taille — **FAIBLE**

**Fichier concerné :** `src/nectar_render/converter/markdown_parser.py:68` — `_IMAGE_INDEX_CACHE`

**Description :** Le cache d'index d'images est un dictionnaire module-level sans limite de taille ni TTL. Dans une session longue traitant de nombreux dossiers d'assets différents, la mémoire pourrait croître indéfiniment.

**Recommandation :** Ajouter une limite de taille (ex : 50 entrées) avec politique LRU, ou un TTL de 30 minutes.

---

#### S-05 : Subprocess pour qpdf — **FAIBLE (bien géré)**

**Fichier concerné :** `src/nectar_render/services/pdf_compression_service.py`

**Description :** L'appel à `qpdf` utilise `subprocess.run` avec une liste de commandes (pas de `shell=True`), ce qui est la bonne pratique. Le timeout est configurable. **Aucune action requise** — c'est un point positif.

### 3.2 Qualité du Code

#### Q-01 : Code mort dans le CLI — **FAIBLE**

**Fichier concerné :** `src/nectar_render/cli.py:152-154`

```python
fmt = args.format.upper()
if fmt == "PDF+HTML":
    fmt = "PDF+HTML"  # No-op
```

**Description :** Cette condition est un no-op. Elle ne fait rien.

**Recommandation :** Supprimer ces 3 lignes.

---

#### Q-02 : Absence de configuration Ruff dans `pyproject.toml` — **MOYEN**

**Fichier concerné :** `pyproject.toml`

**Description :** Le projet utilise Ruff pour le linting et le formatage (CI, README), mais aucune section `[tool.ruff]` n'est définie dans `pyproject.toml`. Cela signifie que Ruff utilise sa configuration par défaut, qui peut varier selon la version installée.

**Recommandation :** Ajouter une section `[tool.ruff]` explicite avec les règles souhaitées :
```toml
[tool.ruff]
target-version = "py310"
line-length = 88

[tool.ruff.lint]
select = ["E", "F", "W", "I", "N", "UP", "B", "SIM"]
```

---

#### Q-03 : `StyleOptions` avec 36 champs — **MOYEN**

**Fichier concerné :** `src/nectar_render/config.py:27-71`

**Description :** La dataclass `StyleOptions` contient 36 champs. Bien que bien organisée, cette classe est difficile à maintenir et à tester. Le pattern de construction dans `app.py:_collect_style()` (70 lignes) est verbeux et fragile.

**Recommandation :** Regrouper les champs en sous-structures :
```python
@dataclass(slots=True)
class TypographyOptions:
    body_font: str = "Segoe UI"
    body_font_size: int = 12
    # ...

@dataclass(slots=True)
class StyleOptions:
    typography: TypographyOptions = field(default_factory=TypographyOptions)
    code: CodeOptions = field(default_factory=CodeOptions)
    # ...
```
**Note :** Ce refactoring est non trivial et devrait être planifié soigneusement pour ne pas casser la compatibilité des presets sauvegardés.

---

#### Q-04 : `app.py` fait 753 lignes — **MOYEN**

**Fichier concerné :** `src/nectar_render/ui/app.py`

**Description :** La classe `NectarRenderApp` est très grande (753 lignes). Bien qu'elle délègue aux panneaux et contrôleurs, elle reste le point central de l'application.

**Recommandation :** Extraire la construction UI (`_build_ui`) dans un module séparé `ui/layout.py`. Cela réduirait `app.py` à ~300 lignes de logique pure.

---

#### Q-05 : `_tk_var_dict()` recréé à chaque appel — **FAIBLE**

**Fichier concerné :** `src/nectar_render/ui/app.py:185-227` et ligne 447

**Description :** La méthode `_tk_var_dict()` crée un nouveau dictionnaire à chaque appel. Elle est appelée plusieurs fois (bindings, state manager). Ce n'est pas un problème de performance mais un gaspillage mémoire mineur.

**Recommandation :** Mettre en cache le résultat ou le créer une seule fois dans `__init__`.

---

#### Q-06 : Pas de type hints sur les fonctions privées dans certains fichiers — **FAIBLE**

**Fichiers concernés :** Plusieurs fichiers, notamment `html_builder.py`, `markdown_parser.py`

**Description :** Les fonctions privées (`_css_color`, `_clamp_int`, etc.) ont des type hints, ce qui est bien. Cependant, certaines fonctions utilitaires dans `panels.py` et `widgets.py` pourraient bénéficier de annotations plus précises.

**Recommandation :** Ajouter `from __future__ import annotations` si absent et annoter les types de retour manquants.

### 3.3 Refactoring

#### R-01 : Mapping fragile dans le CLI — **MOYEN**

**Fichier concerné :** `src/nectar_render/cli.py:36-64`

**Description :** `_state_dict_to_style` utilise un mapping manuel entre les noms de variables Tk et les attributs `StyleOptions`. Si un nom change dans l'UI, le CLI casse silencieusement (les clés manquantes sont simplement ignorées).

**Recommandation :** Créer une fonction de conversion partagée entre l'UI et le CLI, ou utiliser un mécanisme de sérialisation/désérialisation canonique (ex : `dataclasses.asdict` + transformation).

---

#### R-02 : Duplication de la logique de validation entre `state_manager.py` et `html_builder.py` — **MOYEN**

**Fichiers concernés :**
- `src/nectar_render/ui/state_manager.py:30-48` — `_INT_RANGES`, `_DOUBLE_RANGES`
- `src/nectar_render/converter/html_builder.py:87-114` — `_clamp_int`, `_clamp_float` avec valeurs en dur

**Description :** Les plages de validation sont définies à deux endroits différents avec des valeurs légèrement différentes. Par exemple, `image_scale_var` est clampé à `(40.0, 100.0)` dans le state manager mais à `(0.4, 1.0)` dans le HTML builder (après conversion). Cette duplication crée un risque de divergence.

**Recommandation :** Centraliser les plages de validation dans `config.py` et les réutiliser dans les deux modules.

---

#### R-03 : `pdf_compression_service.py` utilise `pypdf` en import dynamique — **FAIBLE**

**Fichier concerné :** `src/nectar_render/services/pdf_compression_service.py`

**Description :** `pypdf` est importé dynamiquement dans la méthode `compress`. Si `pypdf` est listé comme dépendance obligatoire dans `pyproject.toml` (ligne 35), l'import dynamique est inutilement défensif.

**Recommandation :** Si `pypdf` est une dépendance obligatoire, utiliser un import statique en haut du fichier. Sinon, le retirer de `dependencies` et le mettre dans `optional-dependencies`.

---

#### R-04 : `ExportOptions.compression` utilise un forward reference — **FAIBLE**

**Fichier concerné :** `src/nectar_render/config.py:77-79`

**Description :** Le champ `compression` utilise une string `"CompressionOptions"` comme type, alors que `CompressionOptions` est défini après `ExportOptions`. C'est fonctionnel grâce à `from __future__ import annotations`, mais l'ordre des définitions pourrait être réorganisé pour plus de clarté.

**Recommandation :** Déplacer `CompressionOptions` avant `ExportOptions` dans le fichier.

### 3.4 Maintenance

#### M-01 : Pas de verrouillage de dépendances — **ÉLEVÉ**

**Fichier concerné :** `pyproject.toml`

**Description :** Le projet utilise des versions minimales (`markdown>=3.6`, `pygments>=2.18`, etc.) sans fichier de verrouillage. Cela signifie que deux développeurs peuvent avoir des versions différentes des dépendances, avec des risques de régression.

**Recommandation :** Ajouter un fichier de verrouillage :
- `pip-compile` (pip-tools) → `requirements.lock`
- Ou `uv lock` → `uv.lock`
- Ou `poetry.lock` si migration vers Poetry

---

#### M-02 : CI ne teste pas WeasyPrint — **MOYEN**

**Fichier concerné :** `.github/workflows/ci.yml:27`

**Description :** Les tests excluent explicitement les tests nécessitant WeasyPrint (`-m "not requires_weasyprint"`). Cela signifie que le pipeline CI ne valide pas le chemin de conversion PDF, qui est la fonctionnalité principale.

**Recommandation :** Ajouter un job séparé qui installe les dépendances système Pango et exécute les tests WeasyPrint sur Ubuntu (où l'installation est plus simple qu'en Windows).

#### M-07 : Pas de configuration de pré-commit hooks — **FAIBLE**

**Recommandation :** Ajouter un fichier `.pre-commit-config.yaml` avec :
```yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.11.0
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format
```

## 8. Exigences de Sécurité Détaillées

### 8.1 Gestion des secrets

**État actuel :** Aucun secret, token ou clé API n'a été détecté dans le code source. Le projet est une application desktop locale qui ne communique avec aucun service externe. **Aucune action requise.**

### 8.2 Vecteurs d'attaque identifiés

| Vecteur | Statut | Défense actuelle | Recommandation |
|---------|--------|-----------------|----------------|
| XSS via Markdown HTML | ⚠️ Partiel | Bleach disponible mais désactivé | Activer par défaut (S-01) |
| Injection CSS via couleurs/fonts | ✅ Protégé | Regex de validation strict | Aucune |
| Injection de commande via qpdf | ✅ Protégé | Pas de shell=True, liste de commandes | Aucune |
| Traversal de chemin | ✅ Protégé | Validation via `pathlib.Path` | Aucune |
| Déni de service via fichier volumineux | ⚠️ Non traité | Aucun | Ajouter une limite de taille de fichier en entrée |
| Fichiers temporaires non nettoyés | ⚠️ Partiel | Nettoyage en cas d'erreur, mais pas en cas de crash brutal | Utiliser `tempfile.TemporaryDirectory` si possible |

### 8.3 Dépendances et supply chain

Le projet dépend de bibliothèques tierces critiques :
- **WeasyPrint** : moteur de rendu PDF (dépend de GTK/Pango natif)
- **bleach** : sanitisation HTML
- **markdown** : parsing Markdown
- **pypdf** : manipulation PDF

**Recommandation :** Ajouter `pip-audit` ou `safety` au pipeline CI pour détecter automatiquement les vulnérabilités connues dans les dépendances.

## 11. Conclusion

Nectar Render est un projet **bien conçu et bien testé** qui démontre une bonne maîtrise de l'architecture logicielle Python. Les problèmes identifiés sont principalement des améliorations de configuration, de sécurité par défaut et de maintenance à long terme. Aucun problème structurel majeur n'a été détecté.

Les modifications recommandées dans ce plan sont **non invasives** et peuvent être livrées incrémentalement sans modifier le comportement utilisateur (à l'exception de l'activation de la sanitisation HTML par défaut, qui est un changement de sécurité justifié et documentable).

Le projet est prêt pour une montée en maturité vers une version 1.0 stable après application de ces recommandations.
